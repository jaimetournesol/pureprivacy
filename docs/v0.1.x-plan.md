# PurePrivacy v0.1.x improvement plan

Saved 2026-05-06 from a codebase review focused on the **non-technical
operator** use case: a box owner who needs to add their friends and family
to their Matrix install through the wizard, not through SSH.

## Decision: Element (Classic), not Element X, for v0.1

Element X is the future Element is pushing, but for a Tor-only appliance
in v0.1 the trade-offs favor Classic:

1. **Cross-signing UI is more discoverable.** Every operator has to verify
   the MCP bot's device once for E2EE rooms.  Classic surfaces this clearly;
   Element X's verification flow is newer and nested.
2. **Tor track record.** Element Classic over Orbot (Android) and Onion
   Browser (iOS) has years of community recipes.  Element X's sliding sync
   over iOS-Onion-Browser is still rough.
3. **Feature surface.** Classic has more of what power users expect
   (settings, cross-signing reset, room debug).  We don't need Element X's
   speed wins more than we need its rough edges.

Element X stays as a documented alternative.  The relevant docs and wizard
templates point at Classic by default.

## Phase A — Operator UX (the non-technical story)

| Task | What changes |
|------|--------------|
| **A1** | Wizard `/people` page: list users from Synapse admin API; Add/Reset/Remove buttons.  Reuses `admin_cli.py`'s `_admin_session` cache. |
| **A2** | Wizard auth: cookie-gated `/people`, `/pair`, `/rotate-token`.  Login uses the admin password already in `.setup-complete`.  Cookie is HMAC-signed with `/shared/secrets/wizard_session_key`. |
| **A3** | After `/people/add` succeeds, show `share_person.html` — homeserver+username+password QR ready to hand to the new user. |
| **A4** | Copy-to-clipboard buttons on every secret `<pre>` (token, onion, password, recovery key, pair code). |
| **A5** | "Show recovery key" button on `done.html`, gated by re-entering the admin password.  Stops accidental tab-close lockouts. |
| **A6** | Pair-code countdown timer on `pair.html`.  Dead code → "Generate fresh code" button. |

## Phase B — Test coverage gaps

| Task | What changes |
|------|--------------|
| **B1** | CI runs `python -m unittest discover -s wizard/tests -t wizard` + `./scripts/test-features.sh`.  Today neither runs in GitHub Actions. |
| **B2** | `scripts/test-health.sh`: kill postgres, assert synapse health goes red within `start_period + interval × retries`, restart, verify recovery.  Layer 5 in `test-all.sh`. |
| **B3** | `scripts/test-backup.sh`: marker user → backup → wipe → restore → marker present.  Plus negative: marker added after backup → restored backup → marker absent. |

## Phase C — CLI / copy polish

| Task | What changes |
|------|--------------|
| **C1** | `pureprivacy <subcommand> --help` works.  Extract per-command usage from existing inline strings. |
| **C2** | `pureprivacy status` prepends a one-line summary: `N/N healthy · onion: ... · setup: complete`. |
| **C3** | Setup form: "Suggested password (recommended). Click to regenerate." with a regen affordance instead of the ambiguous current copy. |

## Phase D — Out of scope for v0.1.x

- Encrypt admin password at rest under recovery key.  v0.2.
- MCP tool audit log + rate-limit.  v0.2.
- Auto-trust MCP bot device for E2EE rooms.  Already in CHANGELOG for v0.2.
- Onion-served wizard (true mobile admin).  Needs Phase A2 first.

## Sequencing

A1+A2+A3 together (one operator-UX change), then B1+B2+B3 (tests), then
A4–A6 + C1–C3 as polish.  D items deferred.
